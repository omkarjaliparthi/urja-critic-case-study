# Worked Example · End-to-End Critique

> A complete walkthrough of one critique run. The artifact, the rubric, the reference set, the score envelope, the developer's next move. The example is fabricated for case-study purposes — no real customer artifact appears here.

## The setup

A small team is building an astrology study app. They use an LLM to generate illustrated glyphs for the planets. Today's run: a generated SVG of **Mercury in Capricorn** — a glyph the team wants in the app's "transit calendar" surface.

The artifact is generated by **Claude 3.5 Sonnet**. It comes out of the generator as an SVG; the CI workflow rasterizes it via headless Playwright before posting to the Critic.

## The request

```http
POST /api/v1/critic/critique
Host: urja.insightsbyomkar.com
x-api-key: <scoped JWT · critic:v1:use>
content-type: application/json

{
  "rubricId":       "illustrate-glyph",
  "referenceSetId": "observatory-engraved-celestial",
  "subjectHint":    "Mercury (planet) glyph in the observatory-engraved register",
  "output": {
    "kind":   "image-data",
    "dataUri": "data:image/png;base64,iVBORw0KGgoAAAANSUhEUg…"
  },
  "useCache": true,
  "timeoutMs": 60000
}
```

The recorded generator provider for this artifact is `anthropic/claude-3.5-sonnet`. The Critic's provider selector therefore picks an OpenAI judge — `openai/gpt-4o`. The provider non-identity rule is enforced before any tokens are spent.

## The rubric

The team uses the public `illustrate-glyph` rubric. Five axes (illustrative weights — actual production weights are private):

| # | Axis | Weight | What's being scored |
|---|---|---|---|
| 1 | `silhouette-read-at-24px` | 0.25 | Does the glyph read as itself at thumbnail? |
| 2 | `line-weight-hierarchy` | 0.20 | Do interior strokes carry weight tiers? |
| 3 | `construction-clarity` | 0.20 | Is the glyph's geometry classically correct? |
| 4 | `craft-fidelity` | 0.20 | Does it match the observatory-engraved aesthetic? |
| 5 | `semantic-honesty` | 0.15 | Does it read as Mercury and *only* Mercury? |

Each axis carries calibration anchors at scores 2 / 5 / 8 / 10. The judge places the score on the scale by reference to the anchors and emits an `anchorBand` field.

## The reference set

`observatory-engraved-celestial` v1.0.0 — a fabricated set for this example, in the spirit of how a real reference set is shaped:

- 40 plates from late-1800s engraved astronomy texts (all public-domain pre-1929)
- Specific reference glyphs for the 7 classical planets across 4 stylistic registers
- Landmark targets — stroke-weight ratios, glyph-bounds aspect ratio, terminal-taper angles
- `licenseProvenance: public-domain`

The reference set's `subjectKind` is `glyph`. The `illustrate-glyph` rubric declares `subjectKinds: ["glyph"]`. Compatibility check passes.

## The provider call

The rubric runner builds the prompt:

1. Loads the rubric prose framing — "you are a strict illustrator-critic. Use the anchor descriptions as floor posts."
2. Loads the rubric's axes, weights, and anchor descriptions.
3. Loads the reference set's references (text + landmarks; no image attachments in this example).
4. Attaches the rasterized artifact as the artifact-under-review.
5. Specifies the response schema the parser will validate against.

The judge LLM (`openai/gpt-4o`) returns structured JSON. The parser validates and clamps. The response goes into the cache under the full tuple key.

## The cross-provider tag

The envelope's `producedBy` field reads `openai/gpt-4o`. The artifact's recorded `generated_by` (carried in the CI workflow's metadata, not the Critic envelope) reads `anthropic/claude-3.5-sonnet`. The provider non-identity guarantee held — and is visible to anyone who reads the envelope.

The PR comment renders this as the first line:

```
generated by anthropic/claude-3.5-sonnet · graded by openai/gpt-4o
```

That single line is the case study's hook in production form. Every critique, every PR, every audit. The cross-provider policy isn't a slide; it's a comment on a pull request.

## The gate decision

The team's `lucky-critique.yml` workflow has a per-rubric threshold of `weightedTotal >= 7.0` and an axis floor of `5.0`. This run scores **7.4** on the weighted total with no axis below 6.0. The gate **passes** and the PR check turns green.

If the threshold had been 7.5, the gate would have failed and the comment would carry the failings list as the action items. The team would either iterate (apply the auto-applicable fix, regenerate, re-score) or override the gate via the documented `critic-manual-review` label path.

## The auto-refine loop in motion

One failing has `autoApplicable: true` with `proposedTargetValue: 1.8` for the `primary_to_accent_weight_ratio` landmark. The auto-refine loop:

1. Reads the failings list.
2. Filters to `autoApplicable: true`. One failing matches.
3. Updates the manifest: `primary_to_accent_weight_ratio: 1.4 → 1.8`.
4. Triggers the generator with the updated manifest.
5. Rasterizes the regenerated SVG.
6. Re-scores via the Critic.

Iteration 2's envelope (omitted here for brevity) shows `line-weight-hierarchy` lifting from 6.0 to 7.5 and the weighted total from 7.4 to 7.7. The other two failings persist (they're `autoApplicable: false` — they need human attention on the path-rendering pass and the crescent length). The loop halts because no further `autoApplicable` failings remain.

The two structural failings surface to the human review queue with the iteration history attached. A developer reads them, decides whether the path-rendering pass change is in-scope for this PR, and either implements the change or files it as a follow-up.

## What this run evidences

- **Cross-provider grading shipped as a property, not a configuration.** The envelope's provider tag is the audit pointer.
- **Anchored scales produce auditable scores.** Each axis includes the `anchorBand` field so the reasoning can be re-examined.
- **Failings are actionable.** The auto-refine loop runs against numeric failings; structural failings escalate. The developer is never told "score is 6, fix something."
- **Public-domain reference corpus is honored.** The reference set's manifest carries `licenseProvenance: public-domain`. Any customer's legal team can re-trace the corpus.
- **The cache key is honest.** Re-running the same input under the same rubric × refset × provider returns a cache hit; bumping any one of the four invalidates the cached score.

## What this run does NOT show

This worked example is a single critique. Real eval surfaces show their value across:

- **Hundreds of artifacts** — drift in the rubric's mean score over time signals calibration drift.
- **Multiple rubric tracks** — the same engine grades character figures, glyphs, and ambient effects without engine changes.
- **Multiple reference sets per rubric** — the same rubric scored against `great-dane` and `canine-animated` produces *different scores* because the reference register is different. Both are valid; the team's bar is "both clearing 8+."
- **Multiple judges over time** — running the same artifact under both registered providers produces two envelopes with different `producedBy` tags. Calibration-suite work decides whether the two providers are eligible to grade the same rubric.

Those views are observability work — score-history dashboards, calibration drift charts — and they're on the v0.9 roadmap. v0.8 ships the gate; the dashboard ships when the gate has earned developer trust.
