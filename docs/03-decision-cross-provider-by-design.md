# 03 · Decision · Cross-Provider By Design

> The headline decision. If a recruiter reads only one decision document in this case study, this is the one.

## The question

When an LLM grades the output of another LLM, should the judge be allowed to be the same provider as the generator?

The honest 2026 answer most teams give: *"in practice, yes — it's cheaper, fewer keys to manage, and the bias isn't that bad."*

The Urja Critic answers: **no, ever, structurally.** The provider mapping is a one-line policy: `judge.provider !== generator.provider`. The check happens before any tokens are spent. There is no opt-out flag in the public API.

## Why the literature says cross-provider matters

The same-family-bias problem is well-documented:

- **"Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena"** (Zheng et al., 2023) — first widely-cited identification of self-preference and position bias when an LLM judges its own family's output.
- **"Self-Preference in LLM-as-Judge"** (multiple groups, 2024) — replication and extension showing the bias survives prompt-engineering mitigations.
- **Anthropic's evaluation guide** (2024) — explicitly recommends using a different model for grading than for generation when the goal is unbiased eval.
- **OpenAI's evals cookbook** (2024) — same recommendation, embedded in the "human alignment" guidance.

The literature converges. The tooling does not.

## What most teams ship

Single-provider judging — generator and judge on the same provider, often the same model, often the same model *version*. The pattern is the path of least resistance because:

- One API key to manage
- One billing relationship
- Lower per-call cost (no second provider markup)
- Eval frameworks (LangSmith, Braintrust, OpenAI Evals, Anthropic Evals) default this way
- The bias shows up at the margin, not at the median — easy to dismiss

Teams know about the bias. They ship anyway. The Critic's bet is that the right way to fix this is not better prompts or better calibration; it's to make the wrong configuration unbuildable.

## The structural fix

The Urja Critic encodes provider non-identity into the type system:

```ts
// The policy is a one-liner. Note: illustrative — production code lives
// in lib/critic/, not this repo.
function selectJudgeProvider(generatorProvider: ProviderId): ProviderId {
  const candidates = listProviders().filter(p => p !== generatorProvider);
  if (candidates.length === 0) {
    throw new CriticError(
      "NO_OPPOSITE_PROVIDER",
      `Cannot judge output generated by "${generatorProvider}" — no opposite provider registered.`,
    );
  }
  return preferredJudgeFor(generatorProvider) ?? candidates[0];
}
```

Three properties make this structural rather than procedural:

1. **No opt-out.** There is no `allowSameProvider: true` flag. There is no "advanced mode." A caller who wants same-provider grading would have to fork the code.
2. **Fail-closed.** If only one provider is registered (e.g. local dev with one API key), the Critic refuses to grade. The error is loud, not silent.
3. **Audit field.** Every score envelope carries `producedBy` (the judge) alongside the recorded `generated_by` (the generator). The provider mismatch is visible to every downstream consumer.

This is what "encode the policy into the type system" means in practice. The wrong configuration cannot compile, cannot run, and cannot be hidden once shipped.

## The trade-off

Cross-provider grading isn't free.

### Cost · ~2× per-call token spend at eval time

Two providers means two billing relationships and two markups. At scale this matters. The mitigation is **sample-rate vs gate-rate split**:

- **Gate calls** — every PR-merge gate runs cross-provider. These are infrequent and high-stakes; ~2× cost is acceptable insurance.
- **Sample calls** — nightly regression sweeps run cross-provider but at a sampling rate (e.g. 10% of the artifact catalog rotated nightly), so the daily cost is bounded.
- **Inspection calls** — `:read` endpoints (rubric and reference-set listing) are free and have no provider cost.

The Critic does not run cross-provider on every individual generator iteration during refinement — that would be cost-prohibitive in a tight loop. The cross-provider gate runs at the *outer* loop (PR merge, nightly regression). Inner-loop iteration uses cached scores from the last cross-provider gate as the baseline.

### Calibration · judge fluency parity

The judge needs to be competent at the rubric domain. If GPT-4o grades a Sanskrit-grammar rubric and Claude doesn't, swapping which one judges produces inconsistent scores.

The mitigation is the **provider-pair calibration suite**: a fixed set of historical artifacts with known scores. Before a provider becomes eligible to grade a rubric, the Critic re-runs the suite under that provider, measures score drift against the baseline, and accepts the provider for that rubric only if drift is bounded. This is the equivalent of the golden-file test discipline in [Kriya's astronomy engine](https://github.com/omkarjaliparthi/insights-astrology-api-case-study/blob/main/docs/07-test-strategy.md) — a known-answer suite that catches calibration regression on every change.

### Operational · two API keys, two failure modes

Two providers means two outages to plan for. The Critic handles this by treating provider unavailability as a 5xx — the gate fails open into a manual-review queue rather than fails closed and blocks deploys. A judge outage at 3 AM should not be a deploy outage at 3 AM.

## The benefit

### Reproducibility across audits

A customer's legal team can re-run any historical critique and get the same envelope, because the provider tag is in the cache key. A naive same-provider judge whose prompt drifts as the model version updates produces non-reproducible audits over time.

### Defensibility in front of customers

When a customer asks *"how do you avoid grade inflation when both your generator and your judge are Claude?"* the answer is *"we don't. The judge is by policy a different provider than the generator. Here's the envelope; here's the provider tag; here's the source."* That answer ends the conversation.

### Forward-compatibility on new providers

When a new frontier provider ships a competitive model — Anthropic 4, GPT-5, the next Gemini — adding it as a judge candidate is a sibling adapter file. The policy doesn't change. The cross-provider guarantee survives provider churn because it's a *property*, not a *configuration*.

### Marketability of the policy itself

This decision is the case-study hook because it's structural and rare. "Cross-provider by default" is something procurement teams understand at first read. Every other decision in the Critic flows from this one.

## When this would be the wrong call

I should be honest about when the policy doesn't fit:

- **Pure benchmark harnesses** that aren't gating production. Bias matters less when no one is shipping based on the result.
- **Domains where only one provider is competent.** If only Claude can grade your rubric (because the domain is so specialized), forcing a different judge produces noise, not signal. The Critic's calibration suite catches this case and refuses to grade.
- **Cost-sensitive prototype phases.** ~2× the per-call cost is real. If a team is pre-revenue and the eval gate is exploratory, single-provider is a reasonable starting point — *with the migration path to cross-provider documented*.
- **Internal experimentation, not external product.** Internal tools with one user (the builder) face less defensibility pressure than a customer-facing gate.

The Critic doesn't claim cross-provider is universally right. It claims it's right for *production-gated, customer-facing, audit-bearing* eval. Which is the surface most AI-platform-PM roles in 2026 own.

## The implementation lever

The provider-pair calibration suite is the implementation lever that makes cross-provider work. Without it, swapping judges produces score drift and the gate becomes inconsistent. The suite is a separate eval surface — fixed inputs, recorded scores per provider, drift tolerance per rubric. Adding a new provider as a judge candidate is gated by the suite.

The suite is private to the source repo, but its shape is straightforward: ~50 historical artifacts per rubric, scored by every registered provider, with a tolerance band per axis. A provider that exceeds the tolerance on more than N axes is rejected as a judge for that rubric. Recalibration is a manual review; we don't auto-accept calibration drift.

## What this decision evidences

For the Senior PM · AI Platforms lane, this is the load-bearing decision document.

- **Policy-over-configuration thinking** — the policy is encoded into the type system, not the runbook.
- **Defensibility framing** — the customer-legal conversation is part of the design, not an afterthought.
- **Cost-aware policy enforcement** — sample-rate vs gate-rate split is how you make ~2× cost bearable.
- **Calibration discipline** — provider-pair calibration is its own eval surface, not a sidecar.
- **Audit-trail-as-feature** — the provider tag in the envelope is a deliberate feature, not telemetry residue.

## Decision

**Cross-provider grading is the only mode the Critic ships in.** Provider non-identity is enforced in the type system, the calibration suite gates new providers, and the cost is managed with a sample-rate vs gate-rate split. The policy survives every implementation refactor because the policy is the product.
